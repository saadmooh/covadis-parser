## الفروقات الجوهرية

### 1. نظام الإحداثيات (Coordinate System)

- **network 1** يستخدم إحداثيات في النطاق X > 500,000 و Y > 2,000,000 (نظام إحداثيات عالمي UTM تقريباً)
- **network 2** يستخدم إحداثيات محلية مختلفة تماماً: X ≈ -2,363,000 و Y ≈ -1,361,000
- وجود إحداثيات سالبة لا يعني أن البيانات "غير صحيحة" — بل هي محولة أو مسقطة بنظام مختلف
- **الأهم**: جميع طبقات Covadis (`EU 1_Regards`, `EU 1_Canalisations`, إلخ) تحمل نفس **أسماء الطبقات** تماماً في كلا الملفين. الفرق الوحيد هو قيم الإحداثيات نفسها
- الكود القديم كان يستخدم شرط `avgX > 500000 && avgY > 2000000` لتصفية عناصر المقطع الطولي (profile view) — وهذا الشرط يمنع ظهور أي بيانات من network 2

### 2. تسمية المقاطع الطولية (Profile Layers)

- **network 1**: `EU 1_PL_N104_N1_Textes` — تستخدم أسماء عقد تبدأ بـ N
- **network 2**: `EU 1_PL_R19_R18_Textes` — تستخدم أسماء عقد تبدأ بـ R
- مع ذلك، النمط (pattern) واحد: `EU 1_PL_*_Textes`
- الكود القديم كان يعتمد على **قائمة ثابتة** (`['EU 1_PL_N104_N1_Textes', 'EU 1_PL_N135_N63_Textes', 'EU 1_PL_N55_N1_Textes']`) مما يجعله غير قابل لاكتشاف المقاطع الجديدة تلقائياً

### 3. كثافة البيانات (Data Density)

- network 2 يحتوي على 238 طبقة (topographie, bâti, AEP, assainissement, incendie) — معظمها غير ضروري لاستخراج شبكة EU 1
- 329 MB من النص DXF — مكتبة `dxf` تحاول تحليل **كل الكيانات** في الذاكرة مما يسبب خطأ `Out of Memory`
- الحل: استخدام محلل خفيف (lightweight parser) يستخرج فقط الطبقات المطلوبة (EU 1_Regards, EU 1_Canalisations, إلخ) ويتجاهل الباقي

### 4. ترميز الملف (File Encoding)

- `network 1.dxf` و `networks 1+2.dxf` يستخدمان ترميز iso-8859-1 (مباشر، بدون BOM)
- `network 2.dxf` يستخدم `ANSI_1252` وينتهي سطوره بـ `\r\n` (CRLF) — أي أن كل سطر يحمل carriage return في نهايته
- المحلل يجب أن يتعامل مع `\r` بشكل صحيح (استخدام `trimEnd()` أو `trim()`)

## كيف جعلنا الكود عاماً؟

### 1️⃣ اكتشاف المقاطع الطولية تلقائياً (Regex بدلاً من القائمة الثابتة)

**قبل:**
```js
const profileLayers = [
  'EU 1_PL_N104_N1_Textes',
  'EU 1_PL_N135_N63_Textes',
  'EU 1_PL_N55_N1_Textes',
]
```

**بعد:**
```js
const allLayersInFile = new Set(entities.map(e => e.layer).filter(Boolean))
const profileLayers = [...allLayersInFile].filter(l =>
  /^EU\s+1_PL_.*_Textes$/.test(l) || /^Proj\d+\s+.*_PL_.*_Textes$/.test(l)
)
```

بهذا أصبح الكود يكتشف أي مقطع طولي بغض النظر عن تسمية العقد (N أو R أو أي حرف آخر).

### 2️⃣ نظام إحداثيات متكيف (Adaptive Coordinate Filter)

نحسب متوسط إحداثيات جميع رؤوس الأنابيب في الطبقة `EU 1_Canalisations`:

```js
const allVerts = allCanalLines.flatMap(e => e.vertices || []);
const avgAllX = allVerts.reduce((s, v) => s + v.x, 0) / allVerts.length;
const avgAllY = allVerts.reduce((s, v) => s + v.y, 0) / allVerts.length;
const isPlanCoords = avgAllX > 300000 && avgAllY > 1000000;
```

- إذا كانت الإحداثيات في النطاق "العالمي" (X>300k, Y>1M) → نطبق الفلتر `avgX > 500000 && avgY > 2000000` لإزالة عناصر المقطع الطولي
- إذا كانت الإحداثيات محلية (مثل network 2) → **نتخطى الفلتر تماماً** لأن كل عناصر `EU 1_Canalisations` هي عناصر تخطيط (plan view)

### 3️⃣ محلل DXF خفيف للتعامل مع الملفات الكبيرة

مكتبة `dxf` تحاول تحليل كل الكيانات في الذاكرة مرة واحدة — وهذا يسبب OOM لملفات 329 MB.

الحل: `extract_network2.mjs` — محلل مخصص يقرأ السطور واحداً تلو الآخر ويستخرج فقط:
- الكيانات الموجودة على الطبقات المطلوبة (`EU 1_Regards`, `EU 1_Canalisations`, `DN 200`, `assai`, إلخ)
- النصوص الموجودة على طبقات المقاطع الطولية (تطابق `^EU\s+1_PL_.*_Textes$`)

هذا يخفض استخدام الذاكرة من عدة GB إلى حوالي 350 MB (329 MB للنص + 126 KB للكيانات المستخرجة).

### 4️⃣ دعم تحميل JSON للملفات المعالجة مسبقاً

في واجهة الويب (`DxfUploader.jsx`):
- أصبح `accept=".dxf,.json"` يقبل ملفات JSON مباشرة
- للملفات >50 MB يظهر تحذير قبل بدء التحليل
- يمكن معالجة الملفات الكبيرة مسبقاً باستخدام `extract_sewer_data.mjs` ثم تحميل الناتج JSON في المتصفح مباشرة

### 5️⃣ التعامل مع `\r\n` و `\n`

الملفات المختلفة تستخدم نهايات أسطر مختلفة:
- `network 1.dxf`: `\n` فقط (LF)
- `network 2.dxf`: `\r\n` (CRLF)

الحل: استخدام `trim()` بدلاً من `trimEnd()` عند قراءة القيم، لأن `trim()` يزيل المسافات البادئة والتابعة (بما فيها `\r` و `\n`).

## ملخص: القواعد العامة لتحليل أي ملف Covadis

| المشكلة | الحل العام |
|---------|-----------|
| أسماء طبقات المقاطع الطولية مختلفة | استخدام regex `/^EU\s+1_PL_.*_Textes$/` |
| نظام إحداثيات مختلف | كشف آلي: حساب متوسط الإحداثيات ثم اختيار الفلتر المناسب |
| ملف كبير جداً (>100MB) | محلل خفيف يستهدف الطبقات المطلوبة فقط + دعم تحميل JSON |
| نهايات أسطر مختلفة (CRLF vs LF) | استخدام `trim()` لجميع القيم المقروءة من النص |
| طبقات إضافية غير مطلوبة (238 layer) | تحديد `RELEVANT_LAYERS` كمجموعة (Set) للطبقات المهمة فقط |
| تسمية المنهولات مختلفة (N vs R) | عدم الاعتماد على بادئة الحرف — الكود يقبل أي معرف نصي |

## هيكل المحلل الخفيف (Lightweight Parser)

```
قراءة الملف (fs.readFileSync + TextDecoder iso-8859-1)
    │
    ▼
تقسيم النص إلى أسطر (split '\n')
    │
    ▼
البحث عن ENTITIES → داخل ENTITIES فقط نبحث عن الكيانات
    │
    ▼
لكل كيان (INSERT, LWPOLYLINE, MTEXT, TEXT, LINE):
    ├── إذا كانت الطبقة ضمن RELEVANT_LAYERS → حفظ الكيان
    └── إذا كانت الطبقة تطابق ^EU 1_PL_.*_Textes$ والنوع TEXT → حفظ الكيان
    │
    ▼
تجاهل باقي الكيانات (HATCH, 3DFACE, ARC, CIRCLE, ...) → توفير كبير في الذاكرة
    │
    ▼
معالجة الكيانات المخزنة بنفس منطق extract_sewer_data.mjs
```

## الخلاصة

الفرق الرئيسي بين network 1 و network 2 **ليس في أسماء الطبقات** — طبقات Covadis (`EU 1_Regards`, `EU 1_Canalisations`, إلخ) متطابقة تماماً. الفروقات الحقيقية هي:

1. **نظام الإحداثيات**: مختلف تماماً — يجب كشفه تلقائياً
2. **حجم الملف**: 329 MB يتطلب محللاً خفيفاً لتجنب OOM
3. **تسمية العقد في المقاطع الطولية**: N-prefix vs R-prefix — ولكن النمط `EU 1_PL_*_Textes` موحد
4. **نهايات الأسطر**: CRLF vs LF — يجب التعامل مع كليهما

باستخدام هذه التعميمات، أصبح الكود قادراً على تحليل أي ملف Covadis بغض النظر عن نظام الإحداثيات أو حجم الملف أو تسمية العقد.
